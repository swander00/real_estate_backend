# **TRREB RESO Media Integration Implementation Plan**

## **1\. Technical Brief: Media--Property Relationship (RESO 1.7 & TRREB specifics)**

In the RESO Data Dictionary 1.7, **Media** is a separate resource that links to other records (e.g. Property listings, Agents, Offices) via a foreign key. Each media record contains a field `ResourceRecordKey`, which is defined as *"the primary key of the related record from the source resource"*, for example the listing's `ListingKey` in the Property resource[\[1\]](https://bedes.lbl.gov/sites/default/files/RESO%20Dictionary.FinalReview.v.1.5.0%20%2820160515T0000%29%20-%20BEDES%20v2.0%20Mapping%2020161025.pdf#:~:text=The%20primary%20key%20of%20the,For%20example%20the). In TRREB's Web API, the Property listing's unique identifier (`ListingKey`) is used as the `ResourceRecordKey` to group all media belonging to that listing. This means all photo and media entries reference the listing by this key (instead of containing a direct `ListingKey` field), establishing a one-to-many relationship (one listing to many media records). The Media resource also includes a `ResourceName` field indicating which type of resource the media is for (e.g. "Property", "Member", or "Office")[\[2\]](https://docs.youriguide.com/rest/reso.html#:~:text=,see%20ResourceName%20Lookups). In our use case we focus on **Property** media, so `ResourceRecordKey` will correspond to a value in the `common_fields.ListingKey` (the primary key of the listing in our database).

Each media record has its own unique `MediaKey` (the primary key of the media record) and various attributes describing the media (URL, type, category, etc.). **TRREB's implementation uses multiple media records for different renditions of the same image** (e.g. different resolutions or watermarked versions). All variants of the same logical image share a common `MediaObjectID`[\[2\]](https://docs.youriguide.com/rest/reso.html#:~:text=,see%20ResourceName%20Lookups), while each variant has a distinct `MediaKey` (often by appending a suffix like "-t", "-m", "-l", "-nw" to indicate thumbnail, medium, large, or no-watermark versions in the `MediaKey` or URL). For example, a listing photo might have four records: thumbnail, medium, large, and a high-res version; all four will have the same `MediaObjectID` (grouping them as the same photo) but different `MediaKey` values and URLs. This scheme allows the API to provide multiple image sizes, but it means **the same photo will appear multiple times in the Media feed** \-- one per size.

Other RESO standard fields in the Media resource include:

* `MediaURL` \-- the direct URL to the media file (e.g. image link)[\[3\]](https://docs.youriguide.com/rest/reso.html#:~:text=,media%20referenced%20by%20the%20record).  
* `MediaType` \-- the file type or MIME type (e.g. "jpeg", "png", "pdf")[\[4\]](https://docs.youriguide.com/rest/reso.html#:~:text=MediaCategory%20Lookups.%20,media%20referenced%20by%20the%20record).  
* `MediaCategory` \-- the category of media (e.g. "Photo", "Video", "Floor Plan", "Branded Virtual Tour", etc.)[\[5\]](https://docs.youriguide.com/rest/reso.html#:~:text=,pdf). For TRREB, listing photos typically use the "Photo" category.  
* `Order` \-- an integer indicating the display order of the media (where 0 or 1 often denotes the first photo). TRREB follows RESO conventions where the main photo is indicated by the lowest order number.  
* `PreferredPhotoYN` \-- a boolean flag indicating if this media is the preferred photo for the listing[\[6\]](https://docs.youriguide.com/rest/reso.html#:~:text=,listing%20or%20other%20record%20the).  
* `MediaStatus` \-- the status of the media, usually "Active" or "Inactive". An Active status means the media is currently associated with the listing, while "Inactive" indicates the media was removed or is no longer valid for that listing (e.g. a deleted photo).  
* `ModificationTimestamp` \-- a timestamp when the media record was last modified.  
* `MediaModificationTimestamp` \-- a separate timestamp specifically for changes to the media content. In RESO 1.7, **MediaModificationTimestamp is updated when the media object changes**, which may differ from the generic record modification time. TRREB's payload includes both, but typically we will use `MediaModificationTimestamp` as the field to track incremental changes for photos (as it captures when the image itself was last added/updated).  
* `Permission` \-- an array of permission designations for the media. TRREB uses this to control which media can be shown publicly. For example, a photo might have `Permission = ["Public"]` if it's allowed on public sites, or "Private" if it's only for authenticated users/agents. (We'll discuss this further in section 6.)

**No** `ListingKey` **field is directly present in the Media JSON payload** \-- instead, the link is via `ResourceRecordKey`. For example, if a listing's `ListingKey` is "W9002096", all its photo records will have `"ResourceRecordKey": "W9002096"`. To fetch media for a given listing via OData, one would filter by `ResourceRecordKey`. The provided Postman sample shows an example filtering Media by a known listing key: e.g. `$filter=ResourceRecordKey in ('W9002096') and ImageSizeDescription eq 'Largest'` to retrieve all largest-size images for that listing[\[7\]](https://claude.ai/chat/f15e5f7f-96a4-45c1-bc81-a333168cf1b2). In our system, we will enforce referential integrity by linking `property_media.ResourceRecordKey` as a foreign key to `common_fields(ListingKey)`. In summary, the **Property--Media relationship** is one-to-many and realized by matching `ResourceRecordKey` (Media) to `ListingKey` (Property)[\[1\]](https://bedes.lbl.gov/sites/default/files/RESO%20Dictionary.FinalReview.v.1.5.0%20%2820160515T0000%29%20-%20BEDES%20v2.0%20Mapping%2020161025.pdf#:~:text=The%20primary%20key%20of%20the,For%20example%20the). Each media record carries all the metadata needed to use or filter it, and by following RESO 1.7 field names we ensure our schema is standards-compliant.

## **2\. Database Schema: `property_media` Table Definition**

We will create a new `property_media` table in PostgreSQL (Supabase) to store media records in a RESO-compliant structure. The table will use RESO Data Dictionary 1.7 field names and types as column names for clarity. Key design points include a **composite primary key** and a **foreign key reference to listings**, plus indexes to optimize sync operations.

Below is the **exact SQL definition** for the `property_media` table:

CREATE TABLE property\_media (  
    ResourceRecordKey VARCHAR(50) NOT NULL,        \-- references common\_fields.ListingKey  
    MediaKey VARCHAR(100) NOT NULL,               \-- unique key for this media record (incl. variant)  
    MediaObjectID VARCHAR(100) NULL,              \-- groups size variants of the same image  
    MediaURL TEXT NULL,                           \-- direct URL to the media file  
    MediaCategory VARCHAR(50) NULL,               \-- e.g. 'Photo', 'Video', 'Floor Plan'  
    MediaType VARCHAR(50) NULL,                   \-- e.g. 'jpeg', 'pdf' (file MIME type)  
    "Order" INTEGER NULL,                         \-- display order of the media (0 \= first photo)  
    PreferredPhotoYN BOOLEAN NULL,                \-- true if this is the listing's cover photo  
    Permission TEXT\[\] NULL,                       \-- array of permissions (e.g. {'Public','Private'})  
    MediaStatus VARCHAR(20) NULL,                 \-- 'Active' or 'Inactive'  
    MediaModificationTimestamp TIMESTAMPTZ NULL,  \-- last time this media content was modified  
    ModificationTimestamp TIMESTAMPTZ NULL,       \-- last time the record (any field) was modified  
    PRIMARY KEY (ResourceRecordKey, MediaKey),  
    FOREIGN KEY (ResourceRecordKey) REFERENCES common\_fields(ListingKey)  
);  
\-- Add index to speed up delta sync queries by listing and timestamp:  
CREATE INDEX idx\_property\_media\_resrec\_ts   
    ON property\_media(ResourceRecordKey, MediaModificationTimestamp);  
\-- (Optional) Index on MediaModificationTimestamp alone if global timestamp queries are needed

**Explanation:**

* The **primary key is composite** on `(ResourceRecordKey, MediaKey)`. This choice ensures uniqueness per listing: a given media `MediaKey` can appear under different listings (though in practice `MediaKey` is likely globally unique, we combine keys to be safe and to directly use the listing foreign key in the PK). Grouping the PK this way also aligns with how we will upsert (on conflict on these two fields).  
* `ResourceRecordKey` is a text identifier (we allow up to 50 chars which covers typical TRREB listing IDs or GUIDs) and is a **foreign key to** `common_fields(ListingKey)`. This enforces referential integrity: every media record must belong to a known listing in the `common_fields` table.  
* `MediaKey` is text (could be a GUID or string up to 100 chars). It uniquely identifies a specific media record (including a specific size variant).  
* `MediaObjectID` is text used to group variants. If two rows share the same `MediaObjectID`, they represent the same logical image in different sizes[\[2\]](https://docs.youriguide.com/rest/reso.html#:~:text=,see%20ResourceName%20Lookups). We will use this for deduplication logic (section 4).  
* `MediaURL` is stored as TEXT (since URLs can be long). This is the direct link to the media asset[\[3\]](https://docs.youriguide.com/rest/reso.html#:~:text=,media%20referenced%20by%20the%20record). We will store it for use in the application (e.g. to display images).  
* `MediaCategory` and `MediaType` are short text fields describing what the media is[\[5\]](https://docs.youriguide.com/rest/reso.html#:~:text=,pdf)[\[4\]](https://docs.youriguide.com/rest/reso.html#:~:text=MediaCategory%20Lookups.%20,media%20referenced%20by%20the%20record). For example, `MediaCategory='Photo'` and `MediaType='jpeg'` for a standard listing photo, or `MediaCategory='Virtual Tour'` for tour links. Storing these allows filtering if needed (e.g. only photos vs videos).  
* `"Order"` is an integer representing the media's sort order for the listing. We keep the column name quoted as `"Order"` since it's a reserved word. This corresponds to the photo sequence number given by the MLS. The lowest number is the main photo.  
* `PreferredPhotoYN` is a boolean indicating if this is the preferred photo[\[6\]](https://docs.youriguide.com/rest/reso.html#:~:text=,listing%20or%20other%20record%20the).  
* `Permission` is a text array storing any permission labels from the feed (e.g. `{'Public'}` or `{'Private','Office'}` etc). TRREB's media feed can include a `Permission` array field[\[8\]](https://www.tamparealtors.org/index.php?src=reso&srctype=lister&query=1&submenu=PropertySearch&pos=8750,10,59670#:~:text=...%20MediaKey%22%3A%22ba1ececc80e55da016548e664090b9b8,MediaKey), which tells us where the media is allowed to be displayed. We use a text\[\] column to capture all flags. For example, a value of `{'Public'}` means the media can be shown on public websites, whereas `{'Private'}` might indicate it's for internal use only. (Section 6 will cover how we use this.)  
* `MediaStatus` is a short text field for the media's status (e.g. "Active" or "Inactive"). Storing this allows us to flag or filter out media that are inactive (deleted) without immediately removing the row, if we choose a soft-delete approach. Typically, we expect "Active" for current listing photos and "Inactive" for ones that were removed[\[9\]](https://look4homes.ca/#:~:text=Realtors%20in%20Ajax%20and%20Durham,Active).  
* `MediaModificationTimestamp` is a timestamp (with time zone) for when the media content last changed. We will use this for delta synchronization \-- i.e. to know if a media record is new or updated since the last sync. This corresponds to the field provided in the API (often updated when an image is added or changed).  
* `ModificationTimestamp` is also stored (timestamp). This is the generic last update time of the media record (any field). In many cases this may be the same as `MediaModificationTimestamp` unless metadata changed without changing the image. We include it for completeness and potential troubleshooting, but primarily we'll rely on `MediaModificationTimestamp` for detecting content changes.

**Indexes:** We add an index on `(ResourceRecordKey, MediaModificationTimestamp)` to optimize queries that filter or sort media by listing and timestamp. This is particularly useful if we implement delta sync per listing or need to quickly retrieve all new media for a given listing since a certain time. Additionally, this index helps join performance when pulling a listing with its media in order of update time. If we need to query globally for recent media changes (across all listings), we might also create an index on `MediaModificationTimestamp` alone. However, since our replication strategy will often be triggered by listing changes (using `PhotosChangeTimestamp` \-- see section 5), most lookups will be scoped by `ResourceRecordKey`, making the combined index effective.

Overall, this schema is **fully RESO DD 1.7-compliant** in naming and structure. It's designed to accommodate TRREB's specifics (like permission flags and multiple image sizes) while maintaining consistency with the standard. Each column corresponds to a defined Data Dictionary field, ensuring that our system can integrate with RESO-based data consistently.

## **3\. Data Mapping and Upsert Logic (Media API → Database)**

**Mapping the API payload to our schema:** The TRREB RESO API returns media data in JSON format. We will write a mapper that converts each Media JSON object into a row for the `property_media` table. Pseudocode for processing a batch of media records might look like:

for each mediaRecord in API\_response.value:  
    listingId   \= mediaRecord.ResourceRecordKey   // The listing's key (e.g. "W9002096")  
    mediaId     \= mediaRecord.MediaKey            // Unique media record key (string or GUID)  
    objectId    \= mediaRecord.MediaObjectID       // Logical image grouping ID (may be null for non-images)  
    url         \= mediaRecord.MediaURL            // Media file URL  
    category    \= mediaRecord.MediaCategory       // e.g. "Photo", "Video", etc.  
    type        \= mediaRecord.MediaType           // e.g. "jpeg", "pdf"  
    order       \= mediaRecord.Order               // numeric order (int)  
    preferred   \= mediaRecord.PreferredPhotoYN    // boolean  
    permArray   \= mediaRecord.Permission          // array of permission labels  
    status      \= mediaRecord.MediaStatus         // "Active" or "Inactive"  
    mediaModTs  \= mediaRecord.MediaModificationTimestamp // timestamp of media content change  
    modTs       \= mediaRecord.ModificationTimestamp      // timestamp of any change to record

    UPSERT INTO property\_media values (  
         listingId, mediaId, objectId, url,   
         category, type, order, preferred, permArray, status, mediaModTs, modTs  
    )  
    ON CONFLICT (ResourceRecordKey, MediaKey) DO UPDATE   
         \-- If a record with same listingId & mediaId exists, update its fields:  
         SET MediaObjectID \= excluded.MediaObjectID,  
             MediaURL \= excluded.MediaURL,  
             MediaCategory \= excluded.MediaCategory,  
             MediaType \= excluded.MediaType,  
             "Order" \= excluded."Order",  
             PreferredPhotoYN \= excluded.PreferredPhotoYN,  
             Permission \= excluded.Permission,  
             MediaStatus \= excluded.MediaStatus,  
             MediaModificationTimestamp \= excluded.MediaModificationTimestamp,  
             ModificationTimestamp \= excluded.ModificationTimestamp;

**Upsert strategy:** We use PostgreSQL's `ON CONFLICT ... DO UPDATE` to perform an "upsert". The conflict target is our composite primary key `(ResourceRecordKey, MediaKey)`, which uniquely identifies a media record for a listing. This means:

* If a media record (for that listing and with that MediaKey) **does not exist** in the table, the `INSERT` will succeed \-- we get a new row.  
* If it **already exists**, the `DO UPDATE` clause will execute, refreshing all the fields with the latest data from the API payload (e.g. perhaps the URL changed, or its status was updated).

All fields from the payload are mapped into the corresponding columns directly. This ensures the database always has the latest state for each media item. We update even fields like URL or order in case they were modified (though these don't often change, except when photos are re-ordered or replaced, which would come through as new media records or updated order numbers). The `MediaModificationTimestamp` is critical \-- because the API feed is usually *sorted by this timestamp*, we expect incoming records to be in chronological order of changes. Storing this means we can later filter or compare timestamps to see if a record is newer than what we have. (If needed, one could add a **WHERE clause** in the `DO UPDATE` to only update if the incoming `MediaModificationTimestamp` is more recent than the stored one, to avoid touching unchanged records. However, given the feed and our upsert logic, this is usually not necessary \-- we will only be processing new or changed records anyway.)

**Batch processing and grouping:** The Media API can be consumed in two ways \-- (a) continuously pulling new/updated media across all listings (filtered by modification timestamps), and/or (b) on-demand pulling media for specific listings when triggered. In continuous mode, records will come in sorted by time (possibly mixed across listings). We can process them one by one as above. In on-demand mode (say, a whole listing's photos), we will retrieve a set of media records all with the same `ResourceRecordKey`. In that case, it's wise to **group the upsert operations by listing**. For example, if we fetch all media for listing X, we should: upsert each photo for X, then also handle any deletions for X (covered in section 4). Grouping by listing while processing can help because after upserting all of X's media, we can easily run a cleanup for that listing (remove any now-missing photos).

Our pseudocode above focuses on the insert/update of individual records. In practice, we might implement this via an ORM or by generating multi-row `INSERT ... ON CONFLICT` statements for efficiency. We will also **filter out any media not related to property listings** (e.g. if the feed returns agent or office media, we skip those or handle separately) \-- see section 6\.

We will use `MediaModificationTimestamp` for incremental sync logic. For example, our sync process might call the Media endpoint with a filter like:

$filter=MediaModificationTimestamp gt {lastSyncTime} or   
       (MediaModificationTimestamp eq {lastSyncTime} and MediaKey gt '{lastSyncMediaKey}')

and `$orderby=MediaModificationTimestamp,MediaKey` (this mirrors typical delta-sync patterns). The provided replication script uses a similar approach (filter by timestamp and then by key to break ties, ordering by both) to fetch new media in batches[\[10\]](https://claude.ai/chat/f15e5f7f-96a4-45c1-bc81-a333168cf1b2). Each batch of results from the API will be processed through the upsert loop above. The use of `MediaKey` in the tie-break ensures stable ordering if two changes happened at the exact same timestamp. This way, we can resume from the last seen timestamp+key on subsequent sync runs without missing or duplicating records.

**Ensuring data integrity:** Because we have a foreign key constraint, any media with an unknown `ResourceRecordKey` will be rejected. In practice, this shouldn't happen if we only pull media for listings we have (e.g. by syncing Property first, or by filtering ContractStatus=Available and so on). If a media record comes in for a listing we don't have (perhaps because the listing was added concurrently), the insert will fail \-- our strategy is to always sync the Property resource first so that common\_fields is up-to-date, or do a combined fetch. A good practice is to wrap the media upsert in a transaction and log such anomalies; if encountered, one might need to fetch that missing listing record.

Finally, note that **the mapper should handle** `Permission` **arrays and other data types appropriately**. In PostgreSQL, inserting a text array might require casting (e.g. `ARRAY['Public','Private']`), but if using a parameterized query, it will be handled. We must also consider nullability: not all fields will have values (e.g. `MediaObjectID` might be null for non-image media, `Permission` might be null if not provided). Our table allows nulls for those. The upsert will override old values \-- this means if a field becomes null (e.g. if an image's description or something were removed), our update should reflect that as well.

## **4\. Deduplication and Soft Deletion Strategy**

TRREB's approach of providing multiple size variants for the same image can lead to **duplicate logical records** in our table \-- i.e. multiple rows that represent the same photo in different resolutions. Storing every variant could be redundant for many use-cases (especially if we only need one version to display). We have two complementary strategies to handle this:

**a. Preventing duplicates via ingestion filtering:** The simplest way to avoid duplicate entries per image is to **only ingest one size variant** per logical image. For example, we might choose to only store the largest version of each photo (which is typically the highest resolution available to us and often watermarked for compliance). In practice, we can achieve this by adding a filter to our Media API queries such as `ImageSizeDescription eq 'Largest'`[\[7\]](https://claude.ai/chat/f15e5f7f-96a4-45c1-bc81-a333168cf1b2). TRREB's API uses the `ImageSizeDescription` field to label each variant ("Thumbnail", "Large", "Largest", etc.). By querying only for `Largest`, we retrieve just one record per image (presumably the biggest one). This dramatically reduces duplication in the data and ensures we have all photos at a consistent quality for display. The Postman collection example demonstrates this approach, fetching only the `'Largest'` images for a listing[\[7\]](https://claude.ai/chat/f15e5f7f-96a4-45c1-bc81-a333168cf1b2). We will adopt this filter in our sync process. (If needed, we could also choose `'Large'` if performance or size is a concern, but `'Largest'` ensures maximum detail. In many cases "Largest" is the preferred size for web display after applying our own resizing or optimization.) By **ingesting only one variant, we naturally deduplicate** \-- only one row per actual photo will exist in `property_media`. Each `MediaObjectID` will appear only once in the table.

**b. Handling duplicates if multiple variants are stored:** If for some reason we must store multiple variants (for example, perhaps we want to keep the no-watermark version for internal use and the watermarked for public use), we need logic to group and deduplicate when querying or displaying. We can use the `MediaObjectID` field for this[\[2\]](https://docs.youriguide.com/rest/reso.html#:~:text=,see%20ResourceName%20Lookups). All rows with the same `MediaObjectID` represent the same image. We could enforce a constraint or at least ensure via code that we don't show the same `MediaObjectID` twice in a listing's photo gallery. One approach is to prefer one variant as the "primary" and ignore others in presentation. For instance, if both "Large" and "Largest" are stored, we might always choose "Largest" to display and skip "Large" to avoid duplicates. Our table design does not prevent storing duplicates (since the primary key is (listing, mediaKey), not mediaObjectID), but we **use application logic to prevent or filter them**. In summary, if filter (a) is applied, (b) becomes moot. But if not, the application should do `SELECT DISTINCT MediaObjectID` or group by `MediaObjectID` when retrieving images for display, or simply query with `ImageSizeDescription='Largest'`. The recommended practice is to avoid storing extra variants unless truly needed, thereby keeping the database lean and query logic simple.

**Soft delete vs hard delete (handling** `Inactive` **media):** When a listing photo is removed in TRREB, the Media record's `MediaStatus` will change to "Inactive" (and it may stop appearing in the default queries). We need to ensure these removed images don't persist in our application. There are two possible strategies:

* **Hard Delete:** Upon detecting that a media record has been marked Inactive or is no longer present, we remove the row from our table (DELETE it). This keeps our data in sync with the MLS \-- if a photo is deleted by an agent, it will be gone from our system too. We can implement this in the sync process: whenever we encounter a media with `MediaStatus = Inactive`[\[9\]](https://look4homes.ca/#:~:text=Realtors%20in%20Ajax%20and%20Durham,Active), we issue a deletion for that (ResourceRecordKey, MediaKey). Similarly, if we do a full resync for a listing and find that some previously stored media keys are not returned by the API (meaning they were likely deleted and no longer even in the feed), we treat those as inactive and remove them. Hard deletion ensures no stale records linger.

* **Soft Delete:** In some cases, we might prefer to mark the record as deleted rather than actually dropping the row \-- for example, to keep an audit trail or if deletions need to be reviewed. Our schema includes `MediaStatus`, which can serve as a soft-delete flag. We could simply update `MediaStatus` to "Inactive" when we get that signal and possibly set `MediaURL` to NULL (since an inactive image might no longer be accessible). The application would then filter out any `MediaStatus != 'Active'` when showing photos. If we choose this route, an index on MediaStatus or including it in queries would be wise for performance. Soft deleting means the data remains in the table, but we **don't show or use inactive entries** except for logging.

**Recommended approach:** We will implement **hard deletes for truly removed media**, combined with the proactive checks via PhotosChangeTimestamp (see section 5). The rationale is that a deleted photo likely should be completely gone to avoid any confusion (and since we have the authoritative source, we can always re-fetch if needed). Our delta sync will catch most removals: if a photo is deleted, typically its `MediaStatus` flips to Inactive and the `ModificationTimestamp` updates, so it will appear in our feed. We can then delete it. As a safeguard, whenever a listing's `PhotosChangeTimestamp` is updated, we can pull the full current media list for that listing and reconcile: any media in our DB for that listing that is not in the current list should be removed. This double-checks against missed events.

We will ensure to log deletions (e.g. "Photo X for Listing Y removed on \[timestamp\]") for debugging or compliance. Since the foreign key is on listing, deleting media rows will not affect the listing record except that a count of photos (if we maintain one) would change. If transactionally needed, we could update a photo count in `common_fields` as well, but that's optional.

Finally, **duplicate prevention across the whole system**: Because `MediaKey` is likely globally unique (often a GUID), one could use a single-column primary key on MediaKey. We chose (listing, mediaKey) to be explicit. This doesn't normally cause duplicates \-- it's extremely unlikely for two different listings to share a MediaKey. Our approach is conservative and won't harm consistency. In practice, each media row is unique; if any duplicates (exact same MediaKey) ever appeared (due to MLS data issues), the ON CONFLICT would log a duplicate-key but our design would catch it as the same listing anyway.

In summary, **we avoid duplicate image entries by filtering to one variant (Largest)** and by using `MediaObjectID` to group if needed. We **remove stale media** either by deleting the row or marking it inactive, based on the `MediaStatus` from TRREB. This keeps the `property_media` table clean and reflective of the current state of listing photos.

## **5\. Migration and Re-Sync Plan**

Implementing this redesigned `property_media` table in production requires careful migration and an initial population (re-sync) of data, with minimal disruption. Below is a step-by-step plan:

**a. Table Deployment (Migration):** Since the new schema is significantly different from the old `property_media` table, the safest approach is to **create the new table side-by-side** and then switch over, rather than trying to ALTER the existing table with data in place. The migration can proceed as follows:

1. **Create New Table:** Run the `CREATE TABLE property_media ...` SQL (as defined in section 2\) in the database. Do not drop the old table yet. At this point, the new table is empty. Ensure the appropriate indexes (and any constraints/triggers) are created as well. This can be done in a transaction to avoid partial application. Because we chose a different name (the same name, assuming we drop old first, or perhaps use a temp name), you might temporarily name it `property_media_new` if the old exists and rename later. The snippet above assumes we have dropped/renamed the old already, but in a live system, you'd likely do `property_media_new` first.

2. **Backfill Data (Full Re-sync):** Perform a **full retrieval of media records** from the TRREB RESO API to populate the new table. This involves calling the Media OData endpoint for all active listings. There are two ways to do this:

3. **Bulk approach:** Use the Media resource with a filter by time \-- e.g., get all media with `ModificationTimestamp` after a very old date (1970-01-01). Essentially, fetch everything in batches (the replication script can be pointed to media with `lastTimestampValue = 1970-01-01T00:00:00Z` to pull all media records[\[11\]](https://claude.ai/chat/f15e5f7f-96a4-45c1-bc81-a333168cf1b2)). This will retrieve *all* media (photos, etc.) in the system. We should include a filter for `ResourceName = 'Property'` to avoid pulling office/member images, and ideally `ImageSizeDescription = 'Largest'` to limit to one record per image. For example: `GET /odata/Media?$filter=ResourceName eq 'Property' and ImageSizeDescription eq 'Largest'`. This will still return a lot of records (all listing photos), but we can page through using $skip/$top. Each batch is then upserted into the new table.

4. **Per-listing approach:** Iterate through all listings in `common_fields` and for each one call the Media endpoint filtered by that listing's `ResourceRecordKey`. For example: `GET /odata/Media?$filter=ResourceRecordKey eq '{ListingKey}' and ImageSizeDescription eq 'Largest'`. This has more overhead (many calls), but allows parallelism and isolates any problematic listings. Given a large board like TRREB, this might be slower than the bulk approach.

Either way, ensure this process runs during off-peak hours or in a background job, as it could be data-intensive (thousands of listings with multiple photos each). Monitor for API rate limits or quotas. Supabase (Postgres) can handle bulk inserts, but we should batch commits (for instance, commit every few thousand records) to avoid a single huge transaction. During this backfill, the old `property_media` table remains serving the application (if the app is using it). No user-facing changes occur yet.

1. **Verify Data Integrity:** Once the new table is fully populated, run checks to ensure it has all the expected data. For example, count the number of photos per listing in the old vs new table (if the old had that concept) to confirm nothing is missing. Verify that primary keys and foreign keys are correct (no orphan ResourceRecordKeys, etc.). Also spot-check a few listings to ensure the photos match what's in TRREB (maybe by comparing a known listing's photo URLs/ count with the new data). This step is crucial to avoid any data loss when switching.

2. **Swap and Finalize:** Now that `property_media` new is ready, we can switch the application to use it. If we created it under a temporary name, we can drop the old table and rename the new one to `property_media`. Since our foreign key from media references `common_fields(ListingKey)`, and that remains the same, there's no change needed on the common\_fields side. The swap can be done in a transaction to avoid any moment with no table:

* BEGIN; DROP TABLE IF EXISTS property\_media\_old; ALTER TABLE property\_media\_new RENAME TO property\_media; COMMIT;

   Alternatively, if the code can be easily repointed, we could simply update queries to use the new table name (but using the same name is cleaner). Make sure to update any dependent views or functions to account for the new schema.

3. **Update Code & Deploy:** Deploy the updated backend code that uses the new `property_media` structure and mapping logic. This includes the new upsert logic, any API changes, and any queries (for example, if the old design stored media differently, update those parts of the application to use the new table's columns). Because we kept naming consistent with RESO, our code can directly reference fields like MediaURL, etc., which is straightforward.

4. **Monitoring:** After the switch, closely monitor the system. Verify that new listings and photo updates are coming in correctly into `property_media`. Also monitor for any foreign key constraint violations (which would indicate a media for missing listing \-- if that occurs, maybe an order-of-operations issue in sync, which we should adjust by syncing listings first).

**Re-sync considerations:** Once the new table is live, we will use a **delta sync strategy going forward**. Two mechanisms will work in tandem: \- **Incremental Media Sync:** Continuously or periodically poll the /Media endpoint for new changes using the last `MediaModificationTimestamp` seen. This catches new photos or photo updates promptly. For example, run this job every few minutes to keep photos up-to-date. \- **PhotosChangeTimestamp triggers:** The Property resource has a field called `PhotosChangeTimestamp`, which is *"the system-generated timestamp of when the last update or change to the photos for this listing was made"*[\[12\]](https://simplyrets.com/blog/data-dictionary#:~:text=PhotosChangeTimestamp,for%20this%20listing%20was%20made). We already store `common_fields.ModificationTimestamp` for listings, but we should also retrieve and store `PhotosChangeTimestamp` for each listing (if not already in common\_fields). With TRREB, when a listing's photos are changed (added/removed/reordered), its PhotosChangeTimestamp is updated. We can leverage this in our property sync: if a listing's PhotosChangeTimestamp has changed since last seen, that's a signal to refresh that listing's media. An efficient approach is: \- In the regular property sync (which runs on `common_fields`), include `PhotosChangeTimestamp` in the SELECT. \- When processing a property record, compare the incoming PhotosChangeTimestamp to the value stored for that listing. If it's greater (meaning new photo activity), then enqueue a job to **fetch that listing's media** (all of them) from the API and reconcile. This targeted refresh ensures we catch deletions or reordering that the incremental media feed might not fully convey. For example, if a photo was removed, the media feed would mark it inactive (we handle that), but if one was re-ordered, the feed might not send anything if no new photo was added (some systems don't send order changes as modifications). By pulling the full set, we can update the Order fields correctly. It's an additional safety net. \- After fetching, update our stored PhotosChangeTimestamp for that listing (so we don't repeat unnecessarily).

Using PhotosChangeTimestamp in this way is a common practice for MLS data sync[\[13\]](https://docs.mlsgrid.com/api-documentation/api-version-2.0#:~:text=API%20Version%202.0%20,RESO%20expects) \-- essentially, "whenever listing X's photo set changed, go get the latest photos for X." It ensures completeness.

**Downtime and data consistency:** During the migration, there will ideally be **no downtime** or only a brief one. By backfilling first and then swapping, we ensure the app can continue reading from the old table while we load the new one in the background. The final switch can be very quick (a metadata operation). We should temporarily pause any media syncing during the final cut-over to avoid writing to the old table. Once the new table is live, resume syncs pointing to it. Any changes that happened during the backfill window should be picked up by our delta sync (we might do one more sweep of the media feed from the backfill cutoff time to "now" to catch anything missed). Also, any listing that changed photos during the migration will have an updated PhotosChangeTimestamp, so it will get refreshed.

If the old `property_media` had to be kept (for legacy reasons) and cannot be dropped immediately, one strategy is to keep both and gradually phase out references to the old. But given we want a clean break to RESO-compliant design, it's better to migrate and drop the old to avoid confusion.

Finally, as an **operational best practice**, we will perform this in a staging environment first, test thoroughly (especially the re-sync results vs known ground truth), and perhaps do the migration in smaller chunks if the dataset is extremely large (e.g., load media for active listings first, then for historical/off-market if needed). Also ensure to have a backup of the old data (even just a CSV export of the old table or a dump) before dropping it, just in case we need to reference it.

## **6\. TRREB-Specific Access Control & Display Considerations**

TRREB's RESO API introduces some nuances around media access that we must handle to remain compliant with MLS rules and to ensure we display media appropriately in our application. Key considerations include **permissions (public vs. private images)**, **image variant selection for UI**, and handling media for other resource types (Office, Member).

**Public vs Private Media Permissions:** In TRREB (and many MLS feeds), certain images or media are not allowed to be shown on public-facing websites (for example, higher-resolution images or branded virtual tours might be restricted to agent use or internal systems). The `Permission` field on media records indicates this. Common values are "Public" (meaning the media can be displayed openly, e.g. on client websites, IDX feeds) and "Private" (meaning the media is only for users with certain credentials, e.g. an authenticated VOW portal or agent-only). It's possible for a media item to have multiple permission flags (hence an array) \-- for instance, some systems use "IDX" and "VOW" as separate flags. In TRREB's case, we have seen simply "Public" vs "Private\`. Our implementation will:

* **Store the permissions** for each media (we do, in the `Permission` column).  
* **Filter media for display based on permission**. For any consumer-facing application (public website or IDX site), we will only display media that are marked as Public. Media with only Private permission will be either hidden or shown only in restricted contexts. For example, if we build a logged-in dashboard for agents or clients (a VOW portal), we could display private media there, since it's behind authentication and likely allowed. On a public listing page, we must not display private images (doing so could violate MLS rules). Our application logic will query `property_media` and include `... WHERE 'Public' = ANY(Permission)` (assuming Permission array) or filter in code. If a media record has `Permission = {'Private'}` only, we skip it for public viewers.  
* **No URL for Private images:** A detail to note \-- some MLS APIs **do not even provide the** `MediaURL` **for private images unless you explicitly have permission**. For instance, Constellation Labs documentation notes that if you query for private images, the response might omit the URL field[\[14\]](https://docs.cdatalabs.com/#:~:text=Constellation%20Data%20Labs%20,not%20include%20the%20MediaURL%20field). This means if an image is marked private, we might get a record with no URL or an inaccessible URL. In our ingestion, if `MediaURL` is null and Permission is Private, that's a strong signal we can't use that media unless we acquire it through another authorized channel. We will likely just not store such entries (or if stored, they won't have a URL so they are effectively placeholders). Since our filter will target `Public` images, we might not encounter missing URLs often. But our code should defensively handle null URLs (not try to display broken images).

By following the above, we comply with TRREB's access rules. For instance, hi-res photos without watermark might be marked private \-- we will store them only if needed and ensure they aren't unintentionally exposed. We also might choose not to ingest *any* media that are private to avoid clutter (unless the product specifically needs them, say for an agent-only view).

**Image filtering & preferred variant for UI:** As discussed in section 4, we plan to store only the "Largest" version of images (which in TRREB's feed is typically a good quality, possibly watermarked size). For the user interface (web or mobile app), we recommend using this Largest variant for displaying property photos. This simplifies things: we have a single URL per photo to use. In cases where multiple sizes are stored or accessible, a question arises: which size to show? Our recommendation is: \- **Use the largest watermarked image for the main photo gallery on public sites.** The Largest resolution provides the best clarity and by using the watermarked version (if TRREB requires watermarking for public display), we ensure compliance. We can down-scale in the browser or via a CDN as needed for thumbnails or responsive sizes, but starting from the largest gives flexibility. \- If performance is a concern, we could retrieve the "Large" (slightly smaller) variant for faster load, but given modern bandwidth and if we plan to lazy-load images, using the best quality is usually fine. \- We avoid using the "-nw" (no watermark) image on a public site \-- even though it might be highest quality \-- because that likely violates policy unless the site is authenticated. Instead, if we have "-nw" images stored (say for internal use), we segregate their use. Perhaps an agent CMA (comparative market analysis) or internal system could use the un-watermarked images, but the public facing site would stick to watermarked ones.

In short, **show "Public" permitted images at a good resolution (Largest)**. For thumbnails or small displays, we can either use the provided thumbnail from the feed or simply scale down the large image. Since the feed does provide thumbnails ("-t"), we could store them for convenience, but it's often not necessary if we can dynamically generate thumbs or use HTML/CSS scaling (especially since we typically have one variant stored).

**Multiple ResourceName values (Property, Office, Member):** The TRREB API's Media endpoint can return media for different resource types \-- not just property listings. For example, **agent headshots** might appear as media with `ResourceName = 'Member'`, and **office logos** as `ResourceName = 'Office'`. These would use the agent's MemberKey or office's OfficeKey as `ResourceRecordKey`. Our current design of the `property_media` table is specifically intended for listing (Property) media (hence the foreign key ties to ListingKey). To "deal with" these other resource types, we have a few options: \- **Ignore/Exclude them:** If our application only cares about property listing photos (which is often the case for consumer real estate platforms), we can simply **filter them out at ingestion**. This means always including `ResourceName eq 'Property'` in our Media API queries. By doing so, we won't fetch or store Office or Member media in this table at all. This is the simplest approach, and we recommend it unless there is a specific need to display agent/office images in the app. \- **Separate tables:** If we do need to handle agent and office media, a clean approach is to create separate tables (e.g. `office_media` and `member_media`) with similar structure but foreign keys to the Office and Member tables, respectively. This segregates data by type and avoids mixing different foreign keys in one table. The structure could mirror `property_media`. \- **Unified media table:** Alternatively, we could add a `ResourceName` column to our media table and make the foreign key optional or polymorphic. However, PostgreSQL doesn't support a polymorphic foreign key out-of-the-box. We could store all media (property, member, office) in one big table, with `ResourceName` column, and drop the FK constraint (or enforce it via application logic depending on ResourceName). This is more complicated and potentially error-prone. Given that property listing photos are by far the majority and our main interest, keeping them isolated is fine.

So, in practice, we will **filter for Property media only** when populating `property_media`. If in future we want agent photos, we can implement a similar ingestion for `Member` media into a different table (and similarly for Office). Note that if we *don't* filter, and say an agent's photo comes in through our media sync, the foreign key constraint will fail (since that ResourceRecordKey won't exist in common\_fields). This is another reason to include `ResourceName eq 'Property'` in our queries. (The Postman collection's default media query might not include it, but we will add it to be safe.)

**Which images to show in UI:** For each property listing, after our sync, we will have multiple media entries (photos, maybe a virtual tour link, etc.) in `property_media`. We should present them appropriately: \- **Photos:** MediaCategory \= Photo \-- these are the property images. We'll gather all Photo URLs (Public ones) and display them in a gallery/slideshow in the order specified by the `Order` field. The photo with the lowest order number should be shown first (main image). \- **Virtual Tours:** If present (MediaCategory might be "Branded Virtual Tour" or "Unbranded Virtual Tour"), we might not show them as images but as links or embedded if the UI supports (for unbranded tours, maybe an iframe). These typically have MediaURL as a link to a tour, or MediaHTML with embed code. \- **Floor Plans / PDFs:** If MediaCategory is "Floor Plan" or documents, we could provide downloads or separate display sections for those. \- We should heed permissions on each of these as well (some virtual tours might be branded and not allowed publicly, etc., depending on local rules).

**Access Control Summary:** All media queries and usage will comply with TRREB's rules by: \- Not exposing media labeled private to unauthorized users. \- Using watermarked images for public display. \- Only using the MLS-provided URLs (which often are CDN links) \-- we won't attempt to download and host the images ourselves unless required, because those URLs may already handle permission (some systems require certain referrers or tokens \-- but typically, the URL is publicly accessible if the media is allowed public, or might be behind an authorized server if private). \- If needed, apply additional filters like watermark overlays or disclaimers in the UI (some boards require showing the MLS logo or credits when displaying photos \-- ensure we follow any TRREB display guidelines outside the scope of raw data, if any).

In conclusion, our implementation plan ensures that we **store media in a RESO-compliant structure** and maintain integrity with the Property listings. We have a robust sync mechanism using timestamps and triggers to keep photos updated. The design addresses TRREB-specific needs: we avoid duplicative storage of multiple sizes, remove or mark deleted photos, and respect media permissions (only showing what we are allowed to). Additionally, we isolate our scope to property listing media unless expansion is needed for other resource types. By following these guidelines and best practices, the system will reliably fetch, map, and store TRREB media, ready for use in production with accurate and up-to-date listing photos and media content.
